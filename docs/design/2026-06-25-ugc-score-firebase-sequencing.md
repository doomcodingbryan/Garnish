## Problem Statement

Garnish is a two-sided marketplace connecting restaurants with food creators for collabs ("Find the right creator for your restaurant"). It's pre-product: 3 commits, no real users, and the original Supabase backend is dead (DNS failure, project likely paused or deleted). A move to Firebase was being scoped immediately before this session (Supabase auth/client/service surface mapped, the 6-table schema and RLS policies inventoried, the proposal-accept stored procedure identified as the hardest piece to port) but **no Firebase plan has been written yet** — `/Users/bchae/.claude/plans/golden-soaring-moon.md` is a different, already-completed plan (the Supabase-based UGC Score feature from a prior session), not a Firebase migration plan. The question for this session: given everything already built (a UGC Score engine, structured proposal negotiation, an offline dev mode), what's actually worth pushing on next, and is any of this a good idea to keep building?

## What Makes This Cool

The UGC Score: a 0-100 creator score blending Engagement (40%), Reach (25%), Consistency (20%), Breadth (15%), with platform-specific benchmarks so a TikTok creator and a YouTube creator can be compared fairly. Weights renormalize when data is missing instead of failing, and a confidence value reflects API-sourced (YouTube, live) vs. self-reported (Instagram/TikTok, manual) data. The genuine "whoa" moment isn't the dashboard, it's handing a real creator their real number and watching whether it tells them something true about their own content.

## Constraints

- Pre-product, solo builder, learning/side-project framing (not chasing funding or PMF metrics).
- No live backend right now; Firebase migration is the prerequisite to testing anything with a real person.
- Already-built surface area is large for a side project: scoring engine + connectors (YouTube live, IG/TikTok manual), proposal/counter/match negotiation (atomic transaction), full offline dev mode, motion-design landing page.

## Premises

1. **Agreed** — The current UGC Score formula (engagement 40/reach 25/consistency 20/breadth 15) has no audience-authenticity signal (bot/fake-follower detection). 2026 industry research treats that as the #2 priority right after raw engagement. Without it, the score stays vulnerable to the "vanity metric in disguise" criticism it's trying to escape.
2. **Agreed** — Neither "the score" nor "the restaurant niche" alone is the real differentiator. CreatorScore (creatorscore.io) already does engagement/authenticity scoring; Collabstr and Mustard already do restaurant-specific creator discovery. The structured proposal → one-time counter → atomic match transaction is the genuinely uncommon part, and the one worth doubling down on.
3. **Agreed** — Since the builder is the only real test user so far, the fastest way to learn if the score says something true is to run it against a real YouTube channel through the already-live connector, not to keep refining the formula in the abstract.
4. **Agreed** — Reviving the backend (whether that's fixing Supabase or migrating to Firebase) is a real, necessary blocker, not optional polish. No real creator can see a real score without a live database and live auth.

## Approaches Considered

### Approach A: Creator-only score reveal (minimal viable)
Fix Firebase auth + Firestore for a single flow: a creator signs in, connects/enters their YouTube handle, sees their real UGC Score breakdown. Restaurant side, proposal negotiation, and dev-mode mocks stay untouched.
- Effort: S · Risk: Low
- Pros: Ships fastest, tests Premise 3 directly, smallest surface area to debug.
- Cons: Defers the negotiation flow (Premise 2's actual moat) and the full-app outcome the builder explicitly chose.
- Reuses: `src/lib/scoring/*`, the live YouTube connector, `UgcScoreCard`, a trimmed creator onboarding page.

### Approach B: Full marketplace on Firebase (ideal architecture)
Write and execute a fresh Firebase migration plan for both roles end to end — auth, a Firestore data model translated from the existing 6-table Postgres schema, the proposal/accept transaction reimplemented as a Firestore transaction (`runTransaction`, optimistic concurrency with retry, not the row-locking the current Postgres stored procedure uses — see Open Questions for what happens on exhausted retries), Firestore security rules replacing RLS, and the dev-mode mock rewritten against the new client shape (its current interface mirrors a Postgres query builder — `.select()`, `.order()`, `.gte()` — which doesn't carry over mechanically). The YouTube connector's quota usage (Data API v3 defaults to 10,000 units/day; each score computation costs a `channels.list` call plus a `videos.list` call per recent video) needs sizing before the cron-driven refresh job runs unattended, or scoring will silently stop mid-day once quota is exhausted.
- Effort: L, larger than a typical backend swap — this touches every file under `src/lib/supabase/*`, every server action in `src/app/actions/*`, and every page that queries Supabase directly · Risk: Med-High (the transaction port is the single trickiest piece, but security rules and the dev-mode mock rewrite are each their own non-trivial risk, not just implementation details of the transaction)
- Pros: Matches what the builder actually chose (Q3, Premise 4); once done, both the score and the negotiation flow (the real differentiator per Premise 2) are live and testable.
- Cons: Larger surface area before anything is demoable; risk of spending a full session on plumbing (security rules, indexes, auth cookie flow) before reaching a real creator.
- Reuses: the existing business logic and validation as a *reference* (server actions, scoring engine, the documented schema/RLS rules) — the Firestore data model and data-access layer itself are new work, not a drop-in port. There is no existing data to migrate: the Supabase project is unreachable, so this is a fresh start, not a data migration.

### Approach C: Shareable score card (creative/lateral)
On top of Approach A, add a screenshot-able, shareable card view of a creator's score that they'd actually want to post.
- Effort: S-M on top of A · Risk: Low
- Pros: The one feature absent from every named competitor; directly targets "who would say whoa" by making the creator want to show *others*, not just see it themselves; reuses existing Framer Motion component work.
- Cons: Doesn't touch the negotiation/marketplace side; virality is speculative with zero users so far.
- Reuses: Approach A's stack plus `src/components/motion`.

## Recommended Approach

**Approach A first, as the committed milestone — with the rest of Approach B planned as a firm follow-on, not abandoned.** To be precise about what's actually committed right now: the work that gets built and scheduled today is Approach A's scope exactly (Firebase auth, creator profile + platform connections, YouTube connector wiring, score display). The restaurant side, Firestore security rules for those collections, and the proposal/accept transaction port are real, intended next work, not optional — but they are a planned-but-not-yet-scheduled follow-on, not something being built in parallel this round.

This is the specific resolution to the tension named in the Cross-Model Perspective: the subagent's one-weekend scope (creator-only, drop the marketplace) is *operationally* what gets built first, but it is rejected as a *final* answer because the negotiation flow is the actual differentiator per Premise 2 — shipping only the score would mean shipping the one thing competitors already have while permanently deferring the one thing they don't. The difference from the subagent's recommendation is the explicit commitment to continue into the rest of Approach B immediately after, not whether the first milestone looks the same.

Add Approach C (shareable card) as an optional step after the Approach A milestone if time allows. Not committed scope.

## Open Questions

- Which Firestore mechanism replaces the current row-locking stored procedure for proposal acceptance: `runTransaction` with optimistic-concurrency retry, or a Cloud Function. Either way, the atomicity guarantee needs its own dedicated test pass (concurrent-accept race condition, double-counter-offer race) before being trusted — it is not a like-for-like port.
- What user-facing behavior happens when a Firestore transaction exhausts its retries (show an error and let the user retry the action manually? auto-retry server-side N times first?). Not yet decided.
- What happens if the real-creator test produces a low or discouraging score, or the YouTube API key/quota fails mid-test. No fallback plan exists yet — this needs to be written before the test is run, not discovered during it.
- Whether bot/fake-follower detection (Premise 1's gap) gets addressed before or after the first real-creator test — not resolved in this session.
- Who the actual first real-creator tester will be (a friend? the builder's own channel?) — named in this session only as "a food creator," not a specific person.

## Success Criteria

A real creator (not the builder, not a fixture) connects their real YouTube handle through the live Firebase-backed app and can see the full score breakdown: the 0-100 number, tier label, the four component sub-scores (engagement/reach/consistency/breadth), and the confidence label. Falsifiable proxy for "tells them something true": the tester gives written feedback on whether the breakdown matches their own sense of their content, not a vague impression of whether they were impressed.
